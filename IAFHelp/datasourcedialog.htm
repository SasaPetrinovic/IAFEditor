Datasource dialog
   

In this dialog we need to specify all the tables and files which will be used in the program. In a program we can use simultaneously RDB tables and flat files. This option is very important if we want to import/export data from/to another applications or if we want to create temporary flat files for later processing.


Available tables

The list on the left side of the dialog shows all tables defined in the Data Dictionary.
At the beginning of the list, RDB tables are displayed. Classical flat files, if exist, are displayed at the end of the list with text (file) appended to the file name.

Table selection

Click table name and press ">" button

-or-

Double-click table name.

Selected table name appears in the first empty line of the right-side grid. By default,
access mode, is set to READ. If we try to selected already chosen table, the message "Table already selected" appears on the screen.

How to remove chosen table?

·     Click side heading of the grid. Whole line becomes selected.
            Click "<" button

            -or

            Press "Delete" key.

·     Right click side heading of the grid. In context menu select 'Remove table' or 'Remove rest'.


How to change table position in the grid?

Click side heading and drag table name down or up to the desired position.

Selected tables grid

This grid have four columns.

Table name

Table name to be used in the program.

Alias

An alternative name for the table. This is just another, local, name for the table. Maximum alias length is 6 characters.

Qualifiers

Qualifiers give additional information for a table (file).
  
/USE: already_chosen_table
An alternative name for the table.
/SPEC: physical_file_name
Valid for files only. By default, physical file name is recorded in the DD. If we want to open a file with different physical name, but with same record structure, then we use this qualifier. If physical file name contains blanks, use quotation mark or apostrophe as a delimiter.
/MANUAL
FILES: It disables automatic file opening at the beginning of the program. To open such file, we must use an OPEN statement somewhere in the program. Physical file name is defined by DD, or by /SPEC qualifier. If you want to change physical file name at run-time, see Assignment operator (format 4).
RDB TABLES: suppress automatic creation of a temporary table.
/DSN: datasource[;UID=user;PWD=psw]
Valid for database tables only. This qualifier overrides datasource name from the Data Dictionary.
/DSN: SYMBOL=symbol_name

 
Symbol_name is a symbol from Registry, branch ..\RT.
/LIKE: [owner.]origin_table_name
Create and use temporary table. Temporary table name is in "Table Name" column. Table structure is the same as "origin_table_name". "Origin_table_name" is any table loaded into the DD. "Owner" is used for DB2 because of format of run-time DML command.
/OWNER: owner
This value is used to generate UPDATE, WRITE, DELETE_RECORD and default read SQL statement.

If omitted, no owner is inserted into generated SQL statement. This means that DB run-time engine uses default owner to access such tables. Default owner is defined by connection string.

Access mode

Allowed access modes are: READ, UPDATE, WRITE, WRITE_NEW and UPDATE_NO_CREATE.
  
READ
Read (select) allowed only.
UPDATE
Read, insert, update and delete allowed. If file does not exist, an empty file will be created.
WRITE
Read and insert allowed. If file does not exist, an empty file will be created.
WRITE_NEW
Read and insert allowed. All records will be deleted at the beginning of the program. If we use a file, new file will be created with exclusive lock.
UPDATE_NO_CREATE
Valid for files only. Read, insert, update and delete allowed, but file must exist.

Access mode is very important parameter. If we choose READ, we cannot use statements like UPDATE, WRITE and DELETE_RECORD in program code. Also, locking mechanism is different for read_only tables.

Open database manually

By default, run-time engine opens database at the very beginning of the program. Initially, program looks for command line parameter DSN, see Command line parameters. If this parameter does not exist, program checks existence of the symbol DSN_applname in System registry.

If you want to override default behavior, click the 'Open database manually' option and use OPEN statement to connect to the database.

Transaction isolation level

Transaction isolation level defines the level of record locking. According to SQL-92 standard, there are four levels: read uncommitted, read committed, repeatable read and serializable. In ODBC, default is read committed.

Note that some databases do not support all transaction isolation levels.

Connect to multiple datasources at run-time (if it is necessary)

Normally, one program connects to one datasource (database) at the very beginning of the program. If you want to access different databases (for example, MS SQL Server and Oracle) in the same program, check 'Connect to multiple datasources' and use /DSN qualifier for tables from the second database.

Example 1:

table_1
table_2
table_3 /dsn: dsn2;UID=user2;PWD=psw2
table_4 /dsn: dsn2
table_5 /dsn: dsn2

Table_1 and table_2 belong to default connection (the connection defined by application, i.e. by symbol DSN_APPLNAME); table_3 belongs to the connection 2.
Table_4 and table_5 belong to the connection 2. We do not need UID and PWD for these two tables because /dsn: dsn2 denotes just belonging to the second connection (real connection was already established for table_3).


Example 2 (the second connection has no fixed DSN):

table_1
table_2
table_3 /dsn: symbol=another_dsn
table_4 /dsn: symbol=another_dsn
table_5 /dsn: symbol=another_dsn

Table_1 and table_2 belong to default connection (the connection defined by application, i.e. by symbol DSN_APPLNAME).
Table_3 belongs to the connection 2. To establish connection 2, run-time engine reads symbol ANOTHER_DSN from the Registry. Table_4 and table_5 belong to the connection 2, as well.


The two tables from different connections cannot be used in one SQL query.

See Using multiple datasources examples.


Temporary tables

Many databases support temporary tables (MS SQL Server, DB2, Oracle, ...). Temporary tables are usually visible to the current connection only, they do not appear in the system catalog and they are deleted when the user disconnects from the server.

MS SQL Server

Temporary table name starts with one or two number signs (#): one # for local temporary tables and two # for global temporary tables. The table can be created using standard CREATE TABLE statement.

Examples:
#TEMP
##TEMP

Create table #temp (pk int non null, col2 char(10), primary key(pk))
Create table ##temp (pk int non null, col2 char(10), primary key(pk))


DB2

Temporary tables are created using DECLARE GLOBAL TEMPORARY TABLE statement.

Oracle

Temporary tables are created using CREATE GLOBAL TEMPORARY TABLE statement.

Using temporary tables

- Click the first column (Table name) of the first empty line of the grid. Manually enter temporary table name, for example TMPTABLE or #TMP.
- Click the third column (Qualifiers) and enter

/LIKE: origin_table_name

where origin_table_name denotes an existing table whose structure is previously loaded into the DD, for example /LIKE: my_table. Origin_table_name doesn't need to be used in the same program, but its definition must exist in the DD. If you use DB2, the qualifier could be /LIKE: anyowner.anytable.
- set access mode to UPDATE.

Automatic table creation

At run-time, after successful connection to the database server, temporary table will be automatically created.

MS SQL Server

Run-time engine invokes CREATE TABLE statement, ignoring LONG fields (TEXT, NTEXT, IMAGE). If primary key exists, it will be created, too. Additional indexes should be created later in program using EXECSQL CREATE INDEX ... statement.
All temporary tables will be created in the first connection, disabling constructs like this

Select_all * from any_table
            ...
            write #temp
            ...
End_select

ODBC driver for MS SQL Server has SQL_MAX_CONCURRENT_ACTIVITIES parameter set to 1 (maximum number of active statements = 1). SELECT_ALL will be executed via connection #1 and WRITE via connection #2. Because of this, "Invalid object name '#TEMP'" error will be displayed.

To solve this problem, you can use global temporary table, for example ##temp.Global temporary tables in MS SQL Server are visible to any connection:

Select_all * from any_table
            ...
            write ##temp
            ...
End_select


DB2

Run-time engine invokes

DECLARE GLOBAL TEMPORARY TABLE tmptable LIKE [owner.]origin_table ON COMMIT PRESERVE ROWS NOT LOGGED

The /LIKE qualifier for DB2 has the format

/LIKE: [owner.]origin_table_name

To access a DB2 temporary table use SESSION qualifier because DECLARE defines temporary table for the current session:

Select_all * from SESSION.tmptable
            ...
End_select

EXECSQL Insert into SESSION.tmptable ...
EXECSQL Delete from SESSION.tmptable ...
EXECSQL Update SESSION.tmptable

WRITE and UPDATE statements automatically add this qualifier.


Oracle

Default SQL statement is

CREATE GLOBAL TEMPORARY TABLE tmptable (col_def ...) ON COMMIT PRESERVE ROWS

Manual table creation

If you want to override default functionality and explicitly create temporary table, use

/LIKE: origin_table_name /MANUAL

Somewhere in program you have to create temporary table using EXECSQL statement.


Note for report & batch programs

The first table (file) in the grid has special meaning. We call it "main table" because this table defines program cycle.

Also, grid contains one more column: Key. This parameter applies to classic, index-sequential files like Btrieve. Because we currently deal with RDB tables and sequential files, this parameter will not be further explained.
